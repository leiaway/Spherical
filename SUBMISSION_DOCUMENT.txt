1. REFACTOR RATIONALE (ARCHITECTURAL DEBT)
================================================================================

CONSTRAINT/PROBLEM:
Our initial Lovable.dev prototype had all location preference management logic
(loading from database, auto-selecting regions, saving preferences) hardcoded
directly inside Index.tsx across 3 separate useEffect hooks and multiple handlers.

VIOLATIONS:
- Single Responsibility Principle: The Index page component was responsible for
  page rendering (View), database persistence (Data Access), and state management
  (Logic) simultaneously.

- Code Duplication: Location save logic appeared in 3 different places:
  1. Auto-select geolocation handler (line 85-88)
  2. Skip location handler (line 109-112)
  3. Preference loading useEffect (line 56-60)

- Tight Coupling: Database queries directly in component, impossible to test
  location logic without testing entire page.

- Limited Reusability: Location preference logic could not be used in Profile
  page or other components.

IMPACT: Component had 296 lines, mixed concerns made debugging difficult, and
any changes to location preference affected all page rendering logic.

SOLUTION: Applied Abstraction to extract location preference logic into a
reusable custom hook: useLocationPreference().

OLD STRUCTURE:
- src/pages/Index.tsx (Monolithic, 296 lines)
  - Page rendering
  - Geolocation integration
  - Location preference state (3 state variables)
  - Location preference loading (useEffect)
  - Location preference saving (2 locations)
  - Region selection
  - Sidebar components

NEW STRUCTURE:
- src/hooks/useLocationPreference.ts (Focused, 76 lines)
  - Location preference state management
  - Database queries for loading/saving
  - Error handling
  - Loading state management

- src/pages/Index.tsx (Cleaner, 220 lines)
  - Page rendering
  - Geolocation integration
  - Region selection
  - Sidebar components
  (location preference logic now via hook)

FUNDAMENTAL PRINCIPLES APPLIED:
1. Single Responsibility Principle - Each module now has one reason to change
2. Abstraction - Hidden location complexity behind clean hook API
3. Separation of Concerns - Database persistence decoupled from UI
4. DRY (Don't Repeat Yourself) - Eliminated duplicate save logic
5. Dependency Injection - Hook accepts userId as parameter

METRICS:
- Lines of Code: 296 → 220 (-26%)
- useEffect hooks in page: 3 → 1 (-67%)
- Database import references: 1 → 0 (removed from page)
- Duplicated save logic: 3 occurrences → 1 (centralized)
