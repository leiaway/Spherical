# Feature 6.1: Music Shuffle Algorithm Design

## Overview
This document describes the unified algorithm and data schema design for a music shuffle system that prioritizes songs played less frequently, ensuring a non-repetitive user experience.

## Database Schema Design

### Core Tables

#### 1. `tracks` Table (Existing)
The existing tracks table contains global track information:
- `id` (UUID): Primary key
- `play_count` (INTEGER): Global play count (legacy, kept for analytics)
- Other fields: title, artist_id, genre_id, etc.

#### 2. `user_track_plays` Table (New)
Per-user play tracking table that enables personalized shuffle algorithms:

```sql
CREATE TABLE public.user_track_plays (
  id UUID PRIMARY KEY,
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  track_id UUID REFERENCES tracks(id) ON DELETE CASCADE,
  play_count INTEGER NOT NULL DEFAULT 0,
  last_played_timestamp TIMESTAMP WITH TIME ZONE,
  first_played_timestamp TIMESTAMP WITH TIME ZONE,
  created_at TIMESTAMP WITH TIME ZONE,
  updated_at TIMESTAMP WITH TIME ZONE,
  UNIQUE(user_id, track_id)
);
```

**Key Fields:**
- `play_count`: Number of times the user has played this track
- `last_played_timestamp`: When the user last played this track (NULL if never played)
- `first_played_timestamp`: When the user first played this track

**Indexes:**
- `idx_user_track_plays_user_id`: Fast lookup by user
- `idx_user_track_plays_track_id`: Fast lookup by track
- `idx_user_track_plays_last_played`: Optimized for recency queries

**RLS Policies:**
- Users can only view/insert/update their own play records
- Ensures data privacy and security

### Helper Function

#### `increment_user_track_play(user_id, track_id)`
A PostgreSQL function that atomically increments play count and updates timestamps:
- Creates a new record if the track hasn't been played by the user
- Increments `play_count` and updates `last_played_timestamp` if record exists
- Uses `ON CONFLICT` for atomic upsert operation

## Shuffle Algorithm Design

### Mathematical Model

The shuffle algorithm uses a **weighted random selection** approach where each song is assigned a weight based on:
1. **Play Count** (inverse relationship)
2. **Recency** (time since last play)
3. **Never Played Bonus** (songs never played get highest priority)

### Weight Calculation Formula

For a track `i` with:
- `play_count_i`: Number of times user has played this track
- `last_played_i`: Timestamp when user last played this track (NULL if never)
- `current_time`: Current timestamp
- `time_since_last_play`: Seconds since last play (or ∞ if never played)

The weight `w_i` is calculated as:

```
w_i = base_weight_i × recency_factor_i × never_played_bonus_i
```

Where:

#### 1. Base Weight (Inverse Play Count)
```
base_weight_i = 1 / (1 + play_count_i)
```

This ensures:
- Never played songs (play_count = 0): weight = 1.0
- Played once: weight = 0.5
- Played twice: weight = 0.33
- Played 10 times: weight = 0.09
- As play_count → ∞, weight → 0

#### 2. Recency Factor (Time Decay)
```
if last_played_i is NULL:
    recency_factor_i = 1.0  (never played, no penalty)
else:
    hours_since_last_play = (current_time - last_played_i) / 3600
    recency_factor_i = min(1.0, 0.1 + (hours_since_last_play / 24) × 0.9)
```

This ensures:
- Songs played in last hour: recency_factor ≈ 0.1 (very low weight)
- Songs played 12 hours ago: recency_factor ≈ 0.55
- Songs played 24+ hours ago: recency_factor = 1.0 (full weight)
- Never played songs: recency_factor = 1.0 (no penalty)

#### 3. Never Played Bonus
```
if play_count_i == 0:
    never_played_bonus_i = 2.0  (double weight for discovery)
else:
    never_played_bonus_i = 1.0
```

This gives unplayed songs a 2x boost to encourage discovery.

### Complete Weight Formula

Combining all factors:

```
if play_count_i == 0:
    w_i = 1.0 × 1.0 × 2.0 = 2.0
else:
    hours_since = (current_time - last_played_i) / 3600
    recency = min(1.0, 0.1 + (hours_since / 24) × 0.9)
    w_i = (1 / (1 + play_count_i)) × recency × 1.0
```

### Selection Algorithm

1. **Calculate Weights**: For each track in the playlist/collection, calculate `w_i`
2. **Normalize Weights**: Convert to probabilities
   ```
   total_weight = Σ(w_i for all tracks)
   probability_i = w_i / total_weight
   ```
3. **Weighted Random Selection**: Use cumulative distribution function (CDF) to select:
   - Generate random number `r` in [0, 1]
   - Find first track where cumulative probability ≥ r
4. **Repetition Prevention**: Track recently played songs (last N songs) and exclude them from selection if they were played within the last M minutes

### Example Weight Calculations

Consider a playlist with 5 tracks:

| Track | Play Count | Last Played | Hours Since | Base Weight | Recency | Never Played Bonus | Final Weight |
|-------|-----------|-------------|-------------|-------------|---------|-------------------|-------------|
| A     | 0         | NULL        | ∞           | 1.0         | 1.0     | 2.0              | **2.0**     |
| B     | 1         | 2 hours ago | 2           | 0.5         | 0.175   | 1.0              | **0.088**   |
| C     | 3         | 1 day ago   | 24          | 0.25        | 1.0     | 1.0              | **0.25**    |
| D     | 5         | 1 week ago  | 168         | 0.167       | 1.0     | 1.0              | **0.167**   |
| E     | 0         | NULL        | ∞           | 1.0         | 1.0     | 2.0              | **2.0**     |

**Selection Probabilities:**
- Total weight = 2.0 + 0.088 + 0.25 + 0.167 + 2.0 = 4.505
- Track A: 2.0 / 4.505 = **44.4%**
- Track B: 0.088 / 4.505 = **2.0%**
- Track C: 0.25 / 4.505 = **5.5%**
- Track D: 0.167 / 4.505 = **3.7%**
- Track E: 2.0 / 4.505 = **44.4%**

Never-played tracks (A, E) have the highest probability, while recently played tracks (B) have the lowest.

## Implementation Considerations

### Performance Optimization

1. **Batch Weight Calculation**: Calculate weights for all tracks in a single query
2. **Caching**: Cache weights for a session to avoid recalculating on every shuffle
3. **Database Queries**: Use efficient queries with proper indexes
   ```sql
   SELECT 
     t.id,
     t.title,
     COALESCE(utp.play_count, 0) as play_count,
     utp.last_played_timestamp
   FROM tracks t
   LEFT JOIN user_track_plays utp 
     ON t.id = utp.track_id AND utp.user_id = $1
   WHERE t.id = ANY($2::uuid[])
   ```

### Edge Cases

1. **Empty Playlist**: Return empty result
2. **All Songs Never Played**: Equal probability for all (or use other factors like genre diversity)
3. **Very Large Playlists**: Consider sampling a subset for weight calculation
4. **Concurrent Plays**: Use database transactions to prevent race conditions

### Anti-Repetition Mechanisms

1. **Recent Play Exclusion**: Exclude songs played in the last N songs (e.g., last 5)
2. **Time-Based Exclusion**: Exclude songs played within the last M minutes (e.g., 30 minutes)
3. **Session Tracking**: Maintain a session queue of recently played tracks

### Algorithm Parameters (Tunable)

- `never_played_bonus`: 2.0 (can be adjusted)
- `recency_half_life`: 24 hours (time for recency_factor to reach 0.55)
- `min_recency_factor`: 0.1 (minimum weight for recently played songs)
- `recent_play_exclusion_count`: 5 (exclude last N played songs)
- `recent_play_exclusion_minutes`: 30 (exclude songs played in last M minutes)

## Data Flow

### When a Track is Played

1. User selects/plays a track
2. Call `increment_user_track_play(user_id, track_id)`
3. Update session queue (add to recent plays)
4. Optionally update global `tracks.play_count` for analytics

### When Shuffling

1. Fetch all tracks in playlist/collection
2. Fetch user play data for those tracks (single query with JOIN)
3. Calculate weights for each track
4. Apply anti-repetition filters (exclude recent plays)
5. Perform weighted random selection
6. Return shuffled order

## Benefits

1. **Personalized Experience**: Each user gets a shuffle based on their listening history
2. **Discovery**: Unplayed songs are prioritized, encouraging exploration
3. **Variety**: Recently played songs are deprioritized, reducing repetition
4. **Fairness**: All songs have a chance to be played, but less-played songs are favored
5. **Scalability**: Efficient database queries and indexes support large playlists

## Future Enhancements

1. **Genre Diversity**: Add weight adjustments to avoid clustering same genres
2. **Artist Diversity**: Prevent too many consecutive songs from the same artist
3. **Mood/Energy Levels**: Consider track characteristics for smoother transitions
4. **User Preferences**: Allow users to adjust shuffle "randomness" vs "fairness"
5. **Machine Learning**: Use ML models to predict user preferences and adjust weights
